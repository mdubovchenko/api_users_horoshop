# API Users Management System (Symfony)

Це реалізація тестового завдання для компанії Хорошоп. Проєкт являє собою REST API для керування користувачами з автентифікацією через Bearer Token та гнучкою системою ролей.

## 🚀 Вимоги для системи

Для запуску проєкту вам знадобляться:
* **Docker** та **Docker Compose**
* **Make** (бажано, для спрощеного запуску команд)

## 🛠 Запуск проєкту

Проєкт повністю контейнеризований. Щоб підняти інфраструктуру, встановити залежності та підготувати базу даних, виконайте команду:

```bash
make build
```

Ця команда автоматично:
 * Підніме контейнери (PHP 8.2+, MySQL 8.0, Nginx).
 * Встановить Composer-пакети.
 * Виконає міграції бази даних.
 * Завантажить тестові фікстури.

Якщо у вас не встановлено make, виконайте команди з Makefile вручну.

## 🔑 Тестові дані та Ролі
У системі налаштована ієрархія ролей: ROLE_ROOT успадковує всі права ROLE_USER.
Для тестування в базу автоматично завантажуються 10 користувачів.
Ви можете використати їхні токени для перевірки доступу:

| ID | Логін | Роль | Опис |
| :--- | :--- | :--- | :--- |
| **1** | `user_1` | `ROLE_TEST` | **Обмежений доступ:** Аутентифікація проходить, але немає прав на жодну операцію. |
| **2** | `user_2` | `ROLE_ROOT` | **Адміністратор:** Повний доступ, включаючи видалення. |
| **3-10** | `user_x` | `ROLE_USER` | **Користувач:** Доступ до перегляду, створення та оновлення. |

Пароль для всіх користувачів: _123456_
Токени генеруються випадковим чином при кожному завантаженні фікстур.

## 📡 Ендпоінти API
Всі запити повинні містити заголовок Authorization: Bearer <token>.
Всі відповіді повертаються у форматі **JSON**.

| Метод | Шлях | Роль | Опис |
| :--- | :--- | :--- | :--- |
| **POST** | `/v1/api/users` | `ROLE_USER` | Створення нового користувача. |
| **GET** | `/v1/api/users/{id}` | `ROLE_USER` | Отримання даних користувача (включаючи хеш пароля). |
| **PUT** | `/v1/api/users` | `ROLE_USER` | Оновлення даних (ID передається в JSON тілі). |
| **DELETE** | `/v1/api/users/{id}` | `ROLE_ROOT` | Видалення користувача з бази. |

## 🛠 Технічні особливості
1. Валідація: Використовується #[MapRequestPayload] для автоматичної валідації вхідних даних (DTO). Помилки повертаються у форматі JSON з кодом 400.
2. Обробка помилок: Реалізовано ExceptionListener, який перехоплює системні виключення та перетворює їх на зрозумілі JSON-відповіді (404 для неіснуючих роутів, 403 для помилок доступу).
3. Безпека:
   * Паролі зберігаються у хешованому вигляді (BCrypt/Argon2).
   * Реалізована ієрархія ролей у security.yaml.
   * Logs & Cache: Всі помилки та логи доступні всередині контейнера за шляхом /var/www/var/.
